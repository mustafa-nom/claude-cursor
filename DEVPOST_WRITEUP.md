# ClaudeCursor — Devpost Writeup

## Inspiration

AI is evolving so fast that even the people building it are falling behind on the tools they use. Every one of us has felt it — a grandmother trying to book a telehealth appointment in MyChart, a first-generation student opening Xcode for the first time, a designer picking up a new 3D tool, a developer staring at a Kubernetes dashboard they've never used before. The bottleneck is no longer intelligence or effort; it's the time between "I want to do this on a computer" and "I figured out which button to click." Generic YouTube tutorials are frozen in time, documentation assumes you already know the vocabulary to search for, and real human tutors cost money and sleep. The gap stays open, and the people it hurts most are the people who most need the thing on the other side of it.

We wanted to build the friend who sits next to you while you learn — a companion that can actually see your screen, hear your voice, and point at the exact thing you're looking for, in whatever app you happen to be in. Not ChatGPT with a screenshot, not a scripted tutor, not another chatbot. Something that collapses the distance between "I'm stuck" and "now I know," for anyone at any skill level.

## What it does

ClaudeCursor is a macOS menu-bar companion that lives entirely in your status bar — no dock icon, no main window. You hold `ctrl+option`, ask a question in plain voice, and a blue cursor flies across your screen to the answer while Claude narrates what it is through ElevenLabs voice. Under the hood, AssemblyAI streams your voice as you speak, ScreenCaptureKit grabs a screenshot of every monitor, and Claude Sonnet 4.6 takes the transcript and the images as a single vision request. Claude then decides — through native tool use — whether to point at a single element, deploy up to eight colored sub-cursors to label an entire interface at once (`explain_screen_elements`), pop open a persistent markdown answer panel for long-form content, query its own memory of your past sessions, research a topic on the web, or start a full Computer Use automation loop with your consent.

On top of that core loop, ClaudeCursor ships a YouTube lesson extractor that turns any tutorial video into a step-by-step lesson with a picture-in-picture player that auto-seeks to the right timestamp, a tutor mode that watches silently and offers help after three seconds of user idleness, a chat window with a date-bucketed sidebar that groups past sessions by app and even by browser tab, and a personal wiki under `~/Library/Application Support/ClaudeCursor/wiki/` that grows with every session and can be queried later. Every interaction is screen-aware, voice-driven, and tuned to teach rather than replace — the cursor points, the voice narrates, and *you* do the clicking, so skill accumulates instead of atrophying.

## How we built it

The app is ~27,300 lines of Swift across 66 files, written in SwiftUI with AppKit bridged in for the menu-bar `NSStatusItem`, the floating companion `NSPanel`, and the full-screen transparent cursor overlay. The core pipeline is: a listen-only `CGEventTap` at the HID level detects `ctrl+option` globally and never steals focus; `BuddyDictationManager` routes microphone audio into a pluggable transcription provider (`AssemblyAIStreamingTranscriptionProvider` primary with `OpenAIAudioTranscriptionProvider` and `AppleSpeechTranscriptionProvider` as fallbacks); `CompanionScreenCaptureUtility` captures every connected display via ScreenCaptureKit; `ClaudeAPI` streams the vision request as SSE and dispatches tool calls inline through `CompanionToolRegistry`; and `OverlayWindow` animates the blue cursor in bezier arcs to the target coordinates while `ElevenLabsTTSClient` streams the spoken response. An `AdaptiveOutputRouter` classifies every response into one of four surfaces — navigation, lesson, answer, or chat — based on whether the pointing tool fired, whether the content is math/code-heavy, or whether the user is in tutor mode.

All API keys live on a Cloudflare Worker (`worker/src/index.ts`) that fronts Anthropic, ElevenLabs, AssemblyAI, OpenAI Whisper, Tavily, YouTube Innertube, and generic URL fetch. The Worker forwards the `anthropic-beta` header so the Computer Use agent loop (`computer_20251124`) can self-correct by feeding per-step resized screenshots back to Claude, with deny-list enforcement that breaks the loop on first refusal, a `ScreenshotDiffDetector` that uses a perceptual dHash for stuck detection, and cache-aware telemetry that logs the three Anthropic input-token buckets plus output tokens into JSONL per-run files and a rollup log. Long-term memory is a Swift-native on-disk wiki: `ObserverAgent` writes PII-stripped session logs, `SessionCompressor` asks Claude to distill each session into a structured markdown page with a headline and 3–10 durable observations, `WikiPageConsolidator` merges duplicates, and `WikiQueryEngine` retrieves pages with weighted keyword scoring (title 4x, tags 3x, summary 2x, body 0.25x) packed into a character-budgeted context bundle. Persistence is SQLite via `PatternDatabase` for chat segments, tutor nudge rate limiting, and Computer Use analytics; the chat session segmenter uses a 3-minute sandwich-merge rule to clean up noisy context switches and batches K segment-title renames into a single Claude call at session end.

## Challenges we ran into

The hardest problems weren't the AI calls — they were the operating-system plumbing and the long tail of streaming edge cases. A listen-only `CGEventTap` is the only reliable way to catch modifier-only shortcuts like `ctrl+option` globally while the app runs in the background; `NSEvent.addGlobalMonitorForEvents` drops events under load. Creating and invalidating a `URLSession` per AssemblyAI streaming session corrupts the macOS connection pool and produces "Socket is not connected" errors after a few rapid reconnections, so we had to refactor to a single long-lived `URLSession` owned by the provider, not the session. The in-flight Claude tool-use JSON arrives chunked across `input_json_delta` SSE events and only becomes parseable on `content_block_stop`, so we had to write an accumulator that tolerates partial JSON. YouTube's caption API broke mid-2025 when the ANDROID Innertube client was deprecated, so we built an IOS-client-context Innertube request through the Worker with an HTML `ytInitialPlayerResponse` scrape as fallback. Every screenshot sent to the Computer Use API has to be resized to exactly the declared tool geometry (1024×768 / 1280×800 / 1366×768) or Claude's coordinates will miss by pixels — most integrations skip this and wonder why automation misfires.

Getting the agent loop to be *safe* was a separate engineering problem. We landed on a hybrid model: Anthropic's Computer Use API is the default because it self-corrects with fresh per-step screenshots, and our own local one-shot CGEvent dispatcher is demoted to a debug escape hatch behind an Option-key-held submenu. We added a shared `AutomationSafetyPolicy` deny-list for banking and password-manager bundles, made the loop break on Claude's first refusal (so it never nags its way around safety training), wired a perceptual-hash stuck detector to kill runaway loops, and gated every sequence behind a cursor-adjacent consent pill that supports keyboard shortcuts and has its own timeout state. On the UI side, the cursor overlay had to join all Spaces, never steal focus, render across multiple monitors with correct coordinate mapping, accept mouse events *only* inside the consent bubble, and fade in and out smoothly for transient-cursor mode when the user hasn't pinned the overlay. Each of those properties required careful `NSPanel` configuration and a lot of deleted code.

## Accomplishments that we're proud of

We're proud that Claude is *core*, not bolted on: remove Claude and nothing is left. We use Claude in at least seven distinct ways — streaming vision chat, native tool use for structured pointing and panel rendering, Computer Use for agent loops, session compression into the wiki, duplicate-page consolidation, cold-start session recap, and YouTube transcript decomposition. The wiki is a real, stateful long-term memory that grows with use, not a placeholder; sessions get PII-stripped, compressed, indexed, and later retrieved with weighted scoring. The chat sidebar groups past sessions by app *and* by browser tab by reading the active-tab URL from Chromium's AX tree with a 150ms timeout so a wedged browser can't block the main actor, and it never persists full URLs (only hostnames) so session tokens don't leak into the wiki.

We're also proud of the quality signals that don't usually make it into a hackathon demo. The cursor-pill renderer is factored into a shared component (`CursorPillBubble` + `CursorPillTypewriter`) and backed by 24 XCTest snapshot cases at 0.99 precision. The chat session segmenter has Swift Testing coverage across ten behavioral cases including the sandwich-merge rule at tight and wide gaps. There's a CI smoke test that exercises the automation escape hatch end-to-end against Chrome's address bar so it doesn't rot. The Computer Use loop logs schema-versioned JSONL with cache-aware telemetry — the kind of instrumentation a production AI system needs to actually be optimized in the wild, not just demoed on stage. And the whole thing ships with a Cloudflare Worker that keeps every API key off the client, which means downstream deployments (libraries, universities, workforce programs) can subsidize API cost centrally without us rewriting anything.

## What we learned

We learned that building on top of a frontier model is less about prompt engineering and more about *interaction design*. The single biggest lever on user experience turned out to be deciding which output surface a response should render on — and that decision is a pure classification problem that doesn't need another model call. We built a small `AdaptiveOutputRouter` enum that reads a few flags (did the pointing tool fire? is there math or code? is the user in tutor mode?) and chose whether to render on the cursor overlay, the lesson overlay, the answer panel, or the chat transcript. A refactor in April narrowed the "answer" class to math and code only, because anything lighter than that felt heavy-handed as a docked panel. That one routing decision changed how the product feels more than any prompt change we made.

We also learned that safety for a screen-aware, computer-controlling agent is not a feature you add at the end — it's the architecture. Opt-in gates, per-sequence consent, deny-lists, refusal-breaks-the-loop (instead of retry-until-it-complies, which is the silent default of most agent stacks), perceptual-hash stuck detection, before-and-after screenshot logging, and a kill switch all have to exist together or not at all. Letting Claude's refusals actually stop work, rather than being treated as errors to route around, turned out to be the most important safety property in the whole system — and it's trivially easy to get wrong. On the ethics side, we learned that "empowerment over replacement" isn't a slogan; it's an interaction grammar. Pointing preserves motor memory and agency; automating erases both. Making pointing the default — and making automation the explicitly-opted-into exception — is the reason the product teaches instead of just completing tasks.

## What's next for Claude Cursor

The near-term roadmap is concrete. We want to port to Windows via UIAutomation + SendInput so the population that most needs this — older adults, community college students, workforce-retraining learners — isn't gated on macOS hardware. We want to ship a Linux build for accessibility-research and institutional-kiosk deployments. We want to add explicit domain-class detection so that when Claude is narrating in medical, legal, or financial contexts it surfaces a visible "this is not professional advice" framing rather than relying only on Claude's built-in caution. We want to expand the pluggable transcription layer with Whisper-large local so the app works fully offline for users on restricted networks. And we want to invest in the memory wiki: longer-range consolidation, user-editable knowledge, shareable wiki packs for specific tools (Xcode, Figma, QuickBooks, MyChart, USCIS portals) that a new user can install as a starter brain for the tool they're trying to learn.

Longer term, the architecture is designed for institutional deployment. The Cloudflare Worker proxy already decouples user cost from Anthropic's per-token pricing, which means a library system, a community college, a state workforce program, or an immigrant-services nonprofit can self-host the Worker with their own keys and subsidize usage for the population they serve. We want to build the reference deployment for one of those partners, measure real-world outcomes on specific tasks (can more seniors book telehealth visits unaided? can more first-gen students ship a working iOS app in a semester?), and turn those outcomes into the evidence that a screen-aware AI companion is not a productivity tool for the already-advantaged — it's access infrastructure for everyone else. That's the product we actually want to build.
